# class 设计规则
1. 获取属性字段的函数命名为 get_<var_name>.

# 项目目标与开发方式
1. `SysyCC` 的目标是同时支持 `SysY22` 和 `C` 语言，后续设计、实现、测试和文档更新都必须以这个双目标为准，不能只按单一语言子集思考。
2. 后续开发默认采用测试驱动开发。实现新能力、修复 bug、调整语义规则或修改编译流程时，优先补充或更新对应测试，再进行代码实现与回归验证。

# 文件结构相关
1. 头文件前必须要有 #pragma once

# 文档维护规则
1. 每次修改项目结构、模块职责、编译流程、CLI 行为、测试组织方式或输出格式时，必须同步更新 `doc/` 目录中的对应文档。
2. 如果新增了目录、模块或脚本，必须在 `doc/README.md` 的树形结构和模块说明中体现。
3. 如果本次改动会影响开发者理解项目的方式，则必须把文档更新作为本次修改的一部分一起提交。
4. 每次修改类设计、对象所有权、继承关系、模块之间的依赖关系或运行主链时，必须同步更新 `doc/modules/class-relationships.md`。
5. `doc/modules/class-relationships.md` 的正确性是最高优先级要求之一，必须保证其内容和当前代码实现一致，不能接受过期、模糊或错误描述。
6. 如果代码改动会影响类关系说明，则类关系文档的更新不是可选项，而是必须与代码同时完成。
